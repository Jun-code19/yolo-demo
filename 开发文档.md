
# 开发文档

## 项目概述

该项目是一个 YOLO 客户端演示系统，主要功能包括视频流分析、人群分析、数据监听、智能方案管理等。它旨在提供一个端到端的解决方案，从视频源获取数据，进行目标检测和分析，并通过 API 和数据推送服务进行交互。

## 目录结构

以下是项目的主要目录结构及其简要说明：

*   `api/`: 包含后端 API 接口的实现，负责处理客户端请求、数据路由、认证等。
*   `src/`: 包含项目的核心业务逻辑，如数据处理、检测任务运行、监听器实现等。
*   `models/`: 存放 YOLO 模型文件（.pt）。
*   `storage/`: 用于存储检测结果、设备图像、事件等数据。
*   `yolo-client/`: 前端客户端代码，通常是一个基于 Vue.js 的 Web 应用。
*   `docs/`: 存放开发文档、SDK 等相关文件。
*   `部署文件-cpu/` 和 `部署文件-cuda11.7.1-cudnn8/`: 包含不同部署环境（CPU 或 GPU）的 Docker 配置和部署脚本。
*   `__pycache__/`, `new_venv/`, `-p/`: 自动生成的文件或虚拟环境目录。

## API 模块

### `api/auth.py`

*   **功能**: 处理用户认证和授权相关逻辑。
*   **主要内容**:
    *   密码哈希与验证 (`verify_password`, `get_password_hash`)。
    *   用户认证 (`authenticate_user`)。
    *   JWT 访问令牌的创建与验证 (`create_access_token`, `get_current_user`)。
    *   管理员权限检查 (`check_admin_permission`)。

### `api/base_dashboard.py`

*   **功能**: 提供数据大屏所需的各项统计数据和系统资源监控信息。
*   **主要内容**:
    *   **`/dashboard/overview-data` (旧)**: 获取设备、检测事件、配置、人群分析任务、边缘服务器、外部事件的总数概览。
    *   **`/dashboard/crowd-analysis-data`**: 获取各人群分析任务的最新分析结果，包括任务名称、人数统计和最后更新时间。
    *   **`/dashboard/alert-history-data`**: 获取最新的告警历史数据。
    *   **`/dashboard/historical-stats-data`**: 获取过去几天（例如5天）的每日外部事件数量统计。
    *   **`/dashboard/detection-event-data`**: 获取检测事件总数，并按引擎名称统计检测数量。
    *   **`/dashboard/detection-type-data`**: 获取按引擎名称分类的检测事件数据，包括总数、昨天和前天的数量以及同比率（旧版本）。
    *   **`/dashboard/overview-smart-data` (新)**: 提供更全面的数据概览，包括设备、检测配置、人群分析任务、智能方案、智能事件、检测事件和外部事件的数量，以及最新的检测事件列表。
    *   **`/dashboard/historical-smart-data` (新)**: 获取过去几天（例如5天）的每日外部事件、检测事件和智能事件的数量统计。
    *   **`/dashboard/type-smart-data` (新)**: 获取按事件类型分类的统计数据，包括总数、今天、昨天和前天的数量以及同比增长率。
    *   **`/dashboard/system-status`**: 获取实时的系统资源使用情况，包括 CPU、内存、磁盘和 GPU 的使用率、总量和已用量，并根据阈值判断系统状态（正常、警告、危险）。

### `api/base_data_listener.py`

*   **功能**: 提供外部事件的 CRUD (创建、读取、更新、删除) 操作接口，以及相关的图片服务和统计功能。
*   **主要内容**:
    *   **`/data-listeners/images/{image_path:path}`**: 用于获取存储的图片文件，支持解码路径和多种文件查找策略。
    *   **`/data-listeners/events` (GET)**: 获取外部事件列表，支持按配置ID、事件类型、设备ID、SN码、通道ID、引擎ID、时间范围进行过滤，并支持分页。
    *   **`/data-listeners/events/{event_id}` (GET)**: 获取单个外部事件的详细信息。
    *   **`/data-listeners/events/{event_id}` (PUT)**: 更新外部事件的状态、备注等信息，并记录操作日志。
    *   **`/data-listeners/events/{event_id}` (DELETE)**: 删除单个外部事件，同时删除关联的图片文件（如果存在），并记录操作日志。
    *   **`/data-listeners/events/batch-delete` (POST)**: 批量删除外部事件及其关联图片，并记录操作日志。
    *   **`/data-listeners/stats/overview`**: 获取外部事件的统计概览数据，包括今日总事件数、各状态事件数、近1小时事件数和今日报警事件数。

### `api/base_smart_scheme.py`

*   **功能**: 管理智能方案（事件订阅）及其关联的智能事件，提供订阅的创建、查询、更新、删除以及事件的查询、更新、批量处理和统计功能。
*   **主要内容**:
    *   **Pydantic 模型**: 定义了 `SmartSchemeBase`, `SmartSchemeCreate`, `SmartSchemeUpdate`, `SmartSchemeResponse` 用于智能方案的数据校验和响应；`SmartEventBase`, `SmartEventCreate`, `ProcessingRecord`, `SmartEventUpdate`, `SmartEventResponse` 用于智能事件的数据校验和响应。
    *   **`/smart-schemes/manager` (GET)**: 获取智能方案（事件订阅）列表，支持分页、按摄像头ID、事件类型、状态和关键词搜索进行过滤。
    *   **`/smart-schemes/manager` (POST)**: 创建新的智能方案。
    *   **`/smart-schemes/manager/{scheme_id}` (GET)**: 获取单个智能方案的详细信息。
    *   **`/smart-schemes/manager/{scheme_id}` (PUT)**: 更新智能方案的信息。
    *   **`/smart-schemes/manager/{scheme_id}` (DELETE)**: 删除指定的智能方案及其关联事件。
    *   **`/smart-schemes/status`**: 获取所有智能方案的当前状态。
    *   **`/smart-schemes/{scheme_id}/status`**: 获取单个智能方案的当前状态。
    *   **`/smart-schemes/events` (GET)**: 获取智能事件列表，支持分页、按订阅ID、事件类型、状态和时间范围进行过滤，并支持关键词搜索。
    *   **`/smart-schemes/events/{event_id}` (GET)**: 获取单个智能事件的详细信息。
    *   **`/smart-schemes/events/{event_id}` (PUT)**: 更新智能事件的状态、处理结果和备注。
    *   **`/smart-schemes/events/{event_id}` (DELETE)**: 删除单个智能事件。
    *   **`/smart-schemes/events/batch-delete` (POST)**: 批量删除智能事件。
    *   **`/smart-schemes/events/batch-process` (POST)**: 批量将智能事件状态更新为“已处理”。
    *   **`/smart-schemes/events/batch-ignore` (POST)**: 批量将智能事件状态更新为“已忽略”。
    *   **`/smart-schemes/stats/summary`**: 获取智能方案和智能事件的统计概览，包括总数、运行中数量和今日事件数等。
    *   **`/smart-schemes/stats/overview`**: 获取智能事件的详细统计概览，包括总事件数、今日事件数以及按类型分类的报警和智能事件数量。
    *   **`/smart-schemes/cameras`**: 获取摄像头列表，用于智能方案的配置。
    *   **`/smart-schemes/event-types`**: 获取支持的事件类型列表（报警事件、智能事件、设备日志）。
    *   **`/smart-schemes/{scheme_id}/logs`**: 获取指定智能方案的模拟日志数据。

### `api/smart_scheme_routes.py`

*   **功能**: 管理智能方案（事件订阅）及其关联的智能事件，提供订阅的 CRUD 操作、生命周期控制（启动、停止、重启），以及事件的查询、更新、批量处理和统计功能。此文件与 `api/base_smart_scheme.py` 功能相似，但增加了与 `src.smartSchemer` 模块的实际交互来实现订阅的启动、停止和重启，并提供系统状态接口。
*   **主要内容**:
    *   **Pydantic 模型**: 定义了 `SmartSchemeBase`, `SmartSchemeCreate`, `SmartSchemeUpdate`, `SmartSchemeResponse` 用于智能方案；`SmartEventBase`, `SmartEventCreate`, `ProcessingRecord`, `SmartEventUpdate`, `SmartEventResponse` 用于智能事件。
    *   **`/smart-schemes/manager` (GET/POST/PUT/DELETE)**: 提供智能方案（事件订阅）的完整 CRUD 操作，包括列表查询（支持过滤、搜索、分页）、创建、获取详情、更新和删除。
    *   **`/smart-schemes/manager/{scheme_id}/start` (POST)**: 启动指定的智能方案，通过调用 `src.smartSchemer` 模块实现。
    *   **`/smart-schemes/manager/{scheme_id}/stop` (POST)**: 停止指定的智能方案，通过调用 `src.smartSchemer` 模块实现。
    *   **`/smart-schemes/manager/{scheme_id}/restart` (POST)**: 重启指定的智能方案，通过调用 `src.smartSchemer` 模块实现。
    *   **`/smart-schemes/status` (GET)**: 获取所有智能方案的当前状态。
    *   **`/smart-schemes/{scheme_id}/status` (GET)**: 获取单个智能方案的当前状态。
    *   **`/smart-schemes/events` (GET/PUT/DELETE/POST)**: 提供智能事件的查询（支持过滤、搜索、分页）、获取详情、更新、删除、批量删除、批量处理和批量忽略操作。
    *   **`/smart-schemes/stats/summary` (GET)**: 获取智能方案和智能事件的统计概览。
    *   **`/smart-schemes/stats/overview` (GET)**: 获取智能事件的详细统计概览（按类型分类）。
    *   **`/smart-schemes/cameras` (GET)**: 获取可用于智能方案配置的摄像头列表。
    *   **`/smart-schemes/event-types` (GET)**: 获取支持的事件类型列表。
    *   **`/smart-schemes/{scheme_id}/logs` (GET)**: 获取指定智能方案的模拟日志数据。
    *   **`/smart-schemes/system/status` (GET)**: 获取 `SmartSchemer` 模块的系统状态。

### `api/crowd_analysis.py`

*   **功能**: 管理人群分析任务，包括任务的创建、查询、更新、删除、启动、暂停，以及获取可用设备、模型和任务历史数据。
*   **主要内容**:
    *   **Pydantic 模型**: 定义了 `LocationInfo`, `AnalysisJobCreate`, `AnalysisJobUpdate`, `AnalysisJobResponse` 用于人群分析任务的数据校验和响应。
    *   **`/crowd-analysis/jobs` (POST)**: 创建新的人群分析任务。
    *   **`/crowd-analysis/jobs` (GET)**: 获取人群分析任务列表，支持分页，并结合内存中的任务状态返回最新信息。
    *   **`/crowd-analysis/jobs/{job_id}` (GET)**: 获取特定人群分析任务的详细信息，结合内存和数据库状态返回。
    *   **`/crowd-analysis/jobs/{job_id}/delete` (DELETE)**: 删除分析任务及其相关的分析结果，并从运行时服务中移除。
    *   **`/crowd-analysis/jobs/{job_id}/run` (POST)**: 立即执行指定的人群分析任务，并在后台运行。
    *   **`/crowd-analysis/available-devices`**: 获取可用于人群分析的设备列表。
    *   **`/crowd-analysis/jobs/{job_id}/pause` (POST)**: 暂停指定的人群分析任务。
    *   **`/crowd-analysis/jobs/{job_id}/resume` (POST)**: 恢复指定的人群分析任务。
    *   **`/crowd-analysis/jobs/{job_id}/export` (GET)**: 导出分析结果数据（当前仅返回任务信息，实际导出逻辑待实现）。
    *   **`/crowd-analysis/available-models`**: 获取可用于人群分析的模型列表。
    *   **`/crowd-analysis/info-devices`**: 获取设备详情，支持按 `device_ids` 过滤。
    *   **`/crowd-analysis/jobs/{job_id}/history`**: 获取分析任务的历史数据，用于展示趋势图，支持按日期范围过滤。
    *   **`/crowd-analysis/jobs/{job_id}` (PUT)**: 更新现有的人群分析任务，支持修改任务配置并根据 `is_active` 状态同步到运行时服务。
    *   **`/crowd-analysis/model-classes/{model_id}`**: 获取特定模型支持的类别信息。

        *   **`/data-listeners/configs` (GET)**: 获取数据监听器配置列表，支持按类型和启用状态过滤，并分页返回。
        *   **`/data-listeners/configs/{config_id}` (GET)**: 获取单个监听器配置的详细信息，包含运行时状态。

        *   **`/data-listeners/configs/{config_id}` (DELETE)**: 删除监听器配置，同时停止并移除监听器，并取消关联的外部事件。

        *   **`/data-listeners/configs/{config_id}/restart` (POST)**: 重启指定的数据监听器。
    *   **状态和监控接口**:
        *   **`/data-listeners/status` (GET)**: 获取所有监听器的当前状态。
        *   **`/data-listeners/configs/{config_id}/status` (GET)**: 获取单个监听器的当前状态。
    *   **事件查询接口 `events`**:
        *   **`/data-listeners/events` (GET)**: 获取外部事件列表，支持多条件过滤和分页。
        *   **`/data-listeners/events/{event_id}` (GET)**: 获取单个外部事件的详细信息。
        *   **`/data-listeners/events/{event_id}` (PUT)**: 更新外部事件的状态、备注等信息。
        *   **`/data-listeners/events/{event_id}` (DELETE)**: 删除单个外部事件及其关联图片。
        *   **`/data-listeners/events/batch-delete` (POST)**: 批量删除外部事件及其关联图片。
    *   **统计接口 `stats`**:
        *   **`/data-listeners/stats/summary` (GET)**: 获取监听器配置和事件的统计摘要，包括总配置数、启用配置数、按类型分类的配置数、总事件数和今日事件数。
        *   **`/data-listeners/stats/overview` (GET)**: 获取外部事件统计概览数据，包括今日总事件数、各状态事件数、近1小时事件数和今日报警事件数。
    *   **批量操作接口 `batch`**:
        *   **`/data-listeners/batch/start` (POST)**: 批量启动监听器。
        *   **`/data-listeners/batch/stop` (POST)**: 批量停止监听器。

### `api/data_push.py`

### `api/heatmap_routes.py`

*   **功能**: 提供热力图相关的 API 接口，包括地图上传、管理，热力区域的创建、更新、删除，数据源的绑定，以及热力图展板的配置和数据获取。
*   **主要内容**:
    *   **文件上传配置**: 定义了上传文件夹、允许的文件扩展名和最大文件大小。
    *   **Pydantic 模型**: 定义了 `HeatmapMapCreate`, `HeatmapMapResponse`, `HeatmapAreaCreate`, `HeatmapAreaResponse`, `HeatmapBindingCreate`, `HeatmapDashboardConfigModel` 用于热力图数据的数据校验和响应。
    *   **地图管理接口 `maps`**:
        *   **`/heatmap/maps` (GET)**: 获取所有活跃的地图列表。
        *   **`/heatmap/maps` (POST)**: 上传新的地图文件，并创建地图记录，支持图片格式验证和尺寸获取。
        *   **`/heatmap/maps/{map_id}/image` (GET)**: 获取指定地图的图片文件。
        *   **`/heatmap/maps/{map_id}` (DELETE)**: 删除地图及其关联的区域、绑定和图片文件（软删除）。
    *   **区域管理接口 `areas`**:
        *   **`/heatmap/maps/{map_id}/areas` (GET)**: 获取指定地图的所有活跃区域列表，并包含区域绑定的人数统计和最后更新时间。
        *   **`/heatmap/areas` (POST)**: 创建新的热力区域，并计算区域面积。
        *   **`/heatmap/areas/{are-id}` (PUT)**: 更新指定热力区域的信息，如名称、点坐标、颜色、最大容量等，并重新计算面积。
        *   **`/heatmap/areas/{are-id}` (DELETE)**: 删除指定热力区域及其关联的绑定（软删除）。
    *   **数据绑定接口 `bindings`**:
        *   **`/heatmap/bindings` (POST)**: 创建数据绑定，将热力区域与数据源（如人群分析任务）关联，并禁用该区域的旧绑定。
        *   **`/heatmap/bindings/{binding_id}` (DELETE)**: 删除指定的数据绑定（软删除）。
        *   **`/heatmap/areas/{are-id}/bindings` (DELETE)**: 删除指定区域的所有数据绑定（软删除）。
    *   **展板配置接口 `dashboard`**:
        *   **`/heatmap/dashboard/config` (GET)**: 获取当前活跃的展板配置。
        *   **`/heatmap/dashboard/config` (POST)**: 保存新的展板配置，并禁用旧配置。
        *   **`/heatmap/dashboard/data` (GET)**: 获取展板显示数据，包括当前地图信息和所有区域的实时人数统计。
    *   **辅助函数**: `allowed_file`, `ensure_upload_dir`, `calculate_polygon_area` 用于文件验证、目录创建和多边形面积计算。

### `api/logger.py`

*   **功能**: 提供统一的系统日志和检测日志记录功能，方便追踪系统操作和事件。
*   **主要内容**:
    *   **`OPERATION_TYPES`**: 一个字典，用于将英文操作类型（如 `create_device`）映射为更具可读性的中文描述（如“创建设备”）。
    *   **`log_action(db, user_id, action_type, target_id, detail)`**: 记录系统操作日志，包括用户ID、操作类型、目标ID和详细信息。它会根据 `OPERATION_TYPES` 字典自动转换操作类型的中文描述。

### `src/ffmpeg_decoder.py` 系列 (FFmpeg 视频解码器)

本系列文件提供基于 FFmpeg 的视频解码功能，用于从 RTSP 流中实时捕获和解码视频帧。它们设计用于提供稳定的帧流，即使在网络不稳定或流中断的情况下也能尝试自动恢复，并根据部署环境（CPU/Docker/GPU）进行了不同的优化。

*   **`src/ffmpeg_decoder.py` (基础解码器)**:
    *   **功能**: 基础的 FFmpeg 软件解码器，提供核心的帧读取、解码、缓冲区管理和自动重连功能。
    *   **主要内容**:
        *   **`FFmpegDecoder` 类**: 核心解码器类，管理 FFmpeg 进程、帧读取线程和帧缓冲区。
        *   **`_create_ffmpeg_process()`**: 使用 `ffmpeg-python` 库创建 FFmpeg 子进程，进行原始视频帧输出。
        *   **`_read_loop()`**: 核心的帧读取循环，负责从 FFmpeg 进程读取帧数据，并处理重试和重启逻辑。
        *   **`read()`**: 对外提供的接口，兼容 OpenCV 的 `read()` 方法，用于获取最新的解码帧。
        *   **`_adjust_buffer_size()`**: 自适应调整内部帧缓冲区大小，优化实时性和资源使用。

*   **`src/ffmpeg_decoder_docker.py` (Docker 优化解码器)**:
    *   **功能**: 在 `src/ffmpeg_decoder.py` 的基础上，为 Docker 环境进行了优化，提高了在容器化环境中的稳定性和效率。
    *   **主要内容**:
        *   **`FFmpegDecoderDocker` 类**: 继承并扩展了基础解码器功能。
        *   **`_create_ffmpeg_process_subprocess()`**: 新增的备用方法，当 `ffmpeg-python` 失败时，尝试直接使用 `subprocess.Popen` 构建和执行 FFmpeg 命令，提高兼容性。
        *   **`_read_frame_data_docker_v4()`**: 重新设计的帧数据读取方法，采用分块读取、总读取超时、`select.select` 非阻塞检查，并处理不完整帧数据。
        *   **启动优化**: 增加了启动等待和进程状态检查，确保进程稳定。

*   **`src/ffmpeg_decoder_docker_gpu.py` (GPU 加速解码器)**:
    *   **功能**: 一个纯粹为 GPU 硬件加速解码设计的 FFmpeg 视频解码器，利用 CUDA、CUVID 或 QSV 等硬件加速技术，显著提高在支持 GPU 的 Docker 环境中 RTSP 流的解码效率。
    *   **主要内容**:
        *   **`FFmpegGPUDecoder` 类**: 继承并扩展 Docker 优化解码器功能。
        *   **`_detect_gpu_support()`**: 核心功能，通过 FFmpeg 命令检测系统是否支持 CUDA、NVIDIA CUVID 或 Intel QSV 等硬件加速解码器。
        *   **GPU 参数集成**: 在 `_create_ffmpeg_process()` 和 `_create_ffmpeg_process_subprocess()` 中，集成 `-hwaccel cuda` 等 GPU 加速参数。
        *   **严格的 GPU 模式**: 强制要求 GPU 可用，否则启动失败。

*   **`src/listeners/http_listener.py` (HTTP 监听器)**:
    *   **功能**: 作为 `BaseListener` 的一个具体实现，能够以两种模式（Webhook 和轮询）从外部 HTTP 源接收数据。负责处理 HTTP 连接、请求解析、数据标准化，并将接收到的数据转换为统一事件格式供后续处理。
    *   **主要内容**:
        *   **`HTTPListener` 类**: 管理 Webhook 服务器或 HTTP 客户端会话。
        *   **Webhook 模式**: 启动 `aiohttp` Web 服务器，监听 HTTP 请求，处理不同 `Content-Type` 的数据，并支持认证。
        *   **轮询模式**: 定时向目标 URL 发送 HTTP 请求获取数据。
        *   **数据处理**: 增强原始数据，推断消息类型，提取设备 ID，并将数据标准化为 `UnifiedEvent`。

*   **`src/listeners/mqtt_listener.py` (MQTT 监听器)**:
    *   **功能**: 作为 `BaseListener` 的一个具体实现，用于连接 MQTT 代理，订阅指定主题并处理接收到的消息。专注于 MQTT 消息的接收、解析、标准化，并将其转换为统一事件格式。
    *   **主要内容**:
        *   **`MQTTListener` 类**: 管理 MQTT 客户端连接和消息处理。
        *   **连接与订阅**: 建立与 MQTT 代理的连接，配置认证、TLS/SSL，并订阅指定主题。
        *   **消息回调**: 通过 `paho-mqtt` 回调函数接收消息，解码负载，尝试解析 JSON，并放入线程安全队列。
        *   **数据处理**: 增强原始数据，推断消息类型和设备信息，并将数据标准化为 `UnifiedEvent`。

*   **`src/listeners/tcp_listener.py` (TCP 监听器)**:
    *   **功能**: 作为 `BaseListener` 的一个具体实现，用于通过 TCP 协议接收数据。支持作为服务器监听传入连接，或作为客户端主动连接远程服务器。负责处理 TCP 连接、数据接收与解析，并将接收到的数据转换为统一事件格式。
    *   **主要内容**:
        *   **`TCPListener` 类**: 管理 TCP 服务器或客户端连接。
        *   **服务器模式**: 启动 `asyncio` TCP 服务器监听传入连接。
        *   **客户端模式**: 主动连接远程 TCP 服务器，并支持重连。
        *   **数据解析**: 支持 JSON、行分隔和二进制数据格式的解析。
        *   **数据处理**: 将接收到的数据标准化为 `UnifiedEvent`。

*   **`src/run_detection_task.py` (检测任务模块)**:
    *   **功能**: 实现核心的检测任务逻辑，负责从视频源获取帧，利用 YOLO 模型执行目标检测和姿态检测，处理检测结果，并根据配置保存事件、推送数据到外部平台，以及通过 WebSocket 向客户端实时广播检测结果。该模块还集成了目标追踪和智能分析功能。
    *   **主要内容**:
        *   **`DetectionTask` 类**: 核心检测任务类，管理模型加载、摄像机连接、帧读取、检测执行和结果处理。
        *   **模型与视频源**: 支持加载 YOLO 模型（CPU/GPU），连接 RTSP 视频流（OpenCV/FFmpeg 解码器），并处理断线重连。
        *   **检测执行**: 实现动态跳帧优化，执行 YOLO 模型推理，并处理目标追踪和智能分析（行为检测、人数统计）。
        *   **事件处理与推送**: 保存检测事件到数据库，推送检测数据到外部平台，并处理预警机制。
        *   **实时预览**: 通过 WebSocket 向客户端实时广播带检测框的图像和检测结果。
        *   **ROI 管理**: 支持绘制线段或多边形形式的 ROI 区域，并进行区域内人数统计、越线检测等。

*   **`src/smartSchemer.py` (事件订阅管理模块)**:
    *   **功能**: 管理对摄像头主动上报的不同类型数据事件的订阅配置。基于 `NetSDK` 开发包实现真实的设备连接和事件订阅，能够处理报警事件、智能事件以及设备断开/重连等系统日志事件。
    *   **主要内容**:
        *   **`DeviceConnection` 类**: 存储设备连接及订阅相关信息。
        *   **`SmartSchemer` 类**: 核心管理器，负责初始化 `NetSDK`、加载和启动智能方案。
        *   **设备连接与认证**: 登录设备，管理 `loginID` 和 `attach_id`。
        *   **事件订阅**: 启动报警事件监听 (`_start_alarm_listen`) 和智能事件监听 (`_start_smart_listen`)。
        *   **状态管理与重连**: 后台监控设备连接状态，并处理断线重连逻辑。
        *   **事件处理与推送**: 通过 `NetSDK` 回调函数接收并解析事件，创建 `SmartEvent` 并通过 `data_pusher` 推送数据。

*   **`src/tracker.py` (目标追踪模块)**:
    *   **功能**: 实现目标追踪和智能分析的核心逻辑。追踪视频帧中的目标，管理它们的轨迹，并基于预定义的 ROI 区域执行各种智能分析（行为检测、人数统计），触发智能事件。
    *   **主要内容**:
        *   **`ObjectTracker` 类**: 核心目标追踪器，管理目标的生命周期、轨迹和状态。
        *   **目标匹配与更新**: 使用 IOU 和（隐式）匈牙利算法匹配检测结果与现有轨迹，更新追踪器状态，支持轨迹平滑和插值。
        *   **ROI 智能分析**: 支持在指定区域内进行行为检测（进入/离开区域、穿越拌线）和人数统计（区域内人数、人流进出），并触发相应事件。
        *   **可视化**: 在图像上绘制目标边界框、ID、轨迹和智能分析信息。

## `base_data_server.py` (数据服务入口)

*   **功能**: 作为 FastAPI 应用的入口文件，主要负责初始化和配置 API 服务器。它聚合了多个 API 路由模块，设置了 CORS 策略，并处理数据库表的创建。该文件定义了数据服务的基础架构。
*   **主要内容**:
    *   **FastAPI 应用初始化**: 创建 FastAPI 实例，配置应用的标题、描述和版本。
    *   **日志配置**: 配置基本的日志记录。
    *   **数据库表创建**: 在应用启动时，根据 SQLAlchemy ORM 模型创建所有的数据库表。
    *   **CORS 配置**: 添加 `CORSMiddleware` 中间件，支持跨域请求。
    *   **路由包含**: 将多个 API 路由模块（如 `api.routes`、`api.heatmap_routes`、`api.base_smart_scheme` 等）注册到 FastAPI 应用中，并设置统一的 API 前缀 (`/api/v1`)。
    *   **Uvicorn 启动**: 使用 `uvicorn.run()` 启动 FastAPI 应用。

## `base_detect_server.py` (检测服务入口)

*   **功能**: 另一个 FastAPI 应用的入口文件，专注于管理和提供与目标检测相关的服务。它与 `base_data_server.py` 共同构成系统的后端，初始化了检测任务管理器、APScheduler，并集成了数据推送、人群分析、事件订阅、设备监控和数据监听等核心逻辑。
*   **主要内容**:
    *   **FastAPI 应用初始化**: 创建 FastAPI 实例，配置应用信息，并指定 `lifespan` 上下文管理器。
    *   **APScheduler 初始化**: 初始化 `BackgroundScheduler` 作为后台任务调度器。
    *   **日志和 CORS 配置**: 配置日志记录和跨域请求。
    *   **`lifespan` 上下文管理器**: 在应用启动时，执行以下操作：创建数据库表、注册监听器类型、启动设备监控服务、启动数据推送服务、初始化事件订阅管理器、启动所有已启用的检测任务、启动人群分析服务、启动数据监听服务、启动全局清理任务。在应用关闭时，以相反的顺序停止所有服务。
    *   **`DetectionServer` 类**: 管理所有检测任务的生命周期，包括启动/停止检测、任务调度、资源监控、实时预览（WebSocket）和事件清理。
    *   **API 路由**: 定义了多个 API 端点，用于启动/停止检测任务、获取任务状态、手动清理事件、获取系统资源状态、模型加载和 WebSocket 检测预览。
    *   **路由聚合**: 包含了 `api.crowd_analysis`、`api.data_push`、`api.rtsp_server`、`api.data_listener_routes` 和 `api.smart_scheme_routes` 的路由。
    *   **Uvicorn 启动**: 使用 `uvicorn.run()` 启动 FastAPI 应用。
